# **Завдання 3: Обробка великих обсягів даних у системах**

## **Опис завдання:**

Ви — архітектор даних у муніципальній ІТ-компанії, яка впроваджує систему «розумного міста». Система щосекунди отримує події від тисяч сенсорів: дані про трафік, рівень забруднення повітря, температуру, відкриття/закриття громадських паркінгів, сигнали про аварії, показники енергоспоживання тощо.

## **Вимоги до виконання:**

1. У режимі реального часу виявляти критичні події (наприклад, перевищення рівня забруднення, затори, аварії) та негайно передавати їх у відповідні служби.
2. Щогодини формувати аналітичні зведення для міських служб (наприклад, статистика трафіку, споживання енергії, рівень шуму).
3. Зберігати всю історію подій для довгострокового аналізу, прогнозування та міського планування.
4. Забезпечити масштабованість, високу доступність і стійкість до збоїв.
5. Мінімізувати втрату даних і уникати дублювання результатів у критичних підсистемах.
6. Звіт може містити діаграми системи. Їх додавання є ***опціональним***, оскільки основне — щоб ваш звіт надавав вичерпні відповіді та пояснення.

## **Завдання:**

1. **Архітектура системи (40%):**
   - **Побудуйте детальну архітектурну схему** (графічно або у вигляді структурованого тексту), яка включає:
      - **Основні компоненти:** джерела подій (сенсори, IoT-пристрої), транспорт, обробку, зберігання, аналітику, інтерфейс для служб реагування
      - **Розмежування:** потокової (**stream**) та пакетної (**batch**) обробки
      - **Використання розглянутих у лекції технологій:** **Kafka**, **Hadoop**, **Spark**, **HDFS**, **Flink**, **Cassandra** тощо з поясненням ролі кожної
      - **Механізми масштабування та відмовостійкості:** реплікації, партиціонування, контрольні точки тощо
2. **Гарантії обробки (20%):**
   - **Для кожного етапу:** (транспортування, обробка потоків, збереження, формування звітів) визначте, які гарантії доставки/обробки (At Least Once, At Most Once, Exactly Once) ви обираєте та чому
   - **Окремо опишіть, як забезпечити Exactly Once семантику:** для критично важливих підсистем (наприклад, автоматичне реагування на аварії або надсилання тривожних сповіщень)
3. **Відновлення після збоїв (20%):**
   - **Опишіть, як система має поводитись:** у разі відмови одного з компонентів (наприклад, вихід з ладу вузла **Kafka**, **Spark** чи **HDFS**)
   - **Які механізми дозволяють відновити обробку даних:** без втрати чи дублювання? Як відбувається контроль цілісності даних?
4. **Випадки використання / SQL-запити (20%):**
   - **Наведіть по одному прикладу:**
      - **Потокового аналітичного запиту:** (наприклад, виявлення перевищення рівня **CO₂** на перехресті в реальному часі)
      - **Пакетного аналітичного запиту:** (наприклад, підрахунок середнього рівня шуму по районах міста за місяць)
   - **Для кожного поясніть:** чому обраний підхід (**stream/batch**) є оптимальним
